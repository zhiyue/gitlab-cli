# gitlab-cli — Design Spec

- **Date**: 2026-04-17
- **Author**: brainstorming session with user (jabari.dai@gmail.com)
- **Target GitLab version**: Enterprise Edition **14.0.5-ee** (deployed at `gitlab.deepwisdomai.com`)
- **Status**: Draft — pending user review before implementation planning

---

## 1. Background & Motivation

The company runs GitLab EE 14.0.5 (released 2021-07-07, out of support since late 2022). We want a Rust command-line tool that agents (Claude Code, Codex, etc.) can drive via standard `Bash` invocations to read and mutate state on this instance without each agent having to craft raw `curl` calls.

**Why a dedicated tool rather than relying on `glab` or `python-gitlab`**:

- `glab` (official Go CLI) targets current GitLab (17.x) and silently depends on APIs that do not exist in 14.0.5 (e.g., work items GraphQL, newer approval endpoints).
- `python-gitlab` CLI is usable but ships no structured error output, has no bounded exit-code vocabulary, and streams human-oriented text by default — the three things an agent needs most.
- A purpose-built Rust binary is easy to vendor as a single static artifact on developer machines and CI runners.

## 2. Goals / Non-goals

### Goals

1. Agents drive the CLI through `bash -c "gitlab …"` and parse JSON from `stdout`.
2. Coverage of the 12 highest-traffic resource families on GitLab 14.0.5 REST API, plus a raw-API escape hatch for everything else.
3. Stable JSON success schema (GitLab's native responses, passed through) and stable structured error schema on `stderr`.
4. Bounded exit-code vocabulary so an agent can decide "retry vs reconfigure vs give up" without parsing prose.
5. Automatic pagination — a `list` command returns **all** matching records by default (subject to `--limit`), not one page.
6. Single static binary for macOS (x86_64+arm64) and Linux musl (x86_64+arm64).
7. Authentication exclusively through Personal Access Tokens resolved from `--token` flag, `GITLAB_TOKEN` env var, or per-host TOML config.

### Non-goals

- GraphQL support (REST is enough for the 12 chosen resource families on 14.0.5).
- OAuth2 interactive login flows, CI_JOB_TOKEN support, LDAP coupling.
- Multi-version compatibility. This CLI is frozen against 14.0.5-ee semantics; 13.x and 15.x+ are explicitly untested.
- Interactive TUI, shell completions (post-MVP).
- Homebrew / apt / Docker distribution (post-MVP).

### Cross-cutting 14.0.5 API semantics

- **Pagination** is offset-based by default: `page=1`, `per_page=20`, `max per_page=100`. The CLI always internally requests `per_page=100` to minimize roundtrips.
- **Keyset pagination** exists but in 14.0 is effectively limited to `GET /projects?order_by=id` — not worth wiring into core.
- **`Link` header** (RFC 5988, W3C-aligned) is authoritative for `next/prev/last` URLs. The legacy `Links` header is deprecated in 14.0 and will not be parsed.
- **Rate limiting** has no dedicated response header in 14.0; HTTP 429 + `Retry-After` is the sole signal.
- **Error body** shapes observed on 14.0.5:
  - Simple: `{"message": "404 Project Not Found"}` or `{"error": "…"}`
  - Validation: `{"field_name": ["error 1", "error 2"]}` (hash of arrays)
  - The CLI normalizes all three into a single structured error (see §7).
- **Authentication**: the instance accepts `PRIVATE-TOKEN: <PAT>` header. We deliberately support only user-scoped PATs to keep the auth path simple; Project/Group Access Tokens, OAuth flows, and CI_JOB_TOKEN are out of scope.
- **EE-only endpoints** present in 14.0.5 that our 12-class set uses: `merge_request_approvals`, epics-related APIs (deferred; use `api` escape hatch).

## 4. Architecture

### Workspace layout

```
gitlab-cli/
├── Cargo.toml                  # workspace root
├── rust-toolchain.toml         # stable, pinned minor
├── crates/
│   ├── gitlab-core/            # lib crate — no clap, no I/O to terminal
│   │   └── src/
│   │       ├── client.rs       # reqwest wrapper: base_url, TLS, headers, timeout
│   │       ├── auth.rs         # PAT resolution: flag > env > TOML config
│   │       ├── config.rs       # TOML load/save, XDG paths
│   │       ├── error.rs        # GitlabError enum + classification
│   │       ├── page.rs         # Link header parsing + PagedStream<T>
│   │       ├── retry.rs        # exponential backoff + Retry-After
│   │       ├── throttle.rs     # token-bucket client-side rps cap
│   │       └── resources/
│   │           ├── projects.rs
│   │           ├── groups.rs
│   │           ├── merge_requests.rs
│   │           ├── issues.rs
│   │           ├── pipelines.rs
│   │           ├── jobs.rs
│   │           ├── commits.rs
│   │           ├── branches.rs
│   │           ├── tags.rs
│   │           ├── files.rs
│   │           ├── repos.rs
│   │           ├── users.rs
│   │           ├── labels.rs
│   │           ├── notes.rs
│   │           └── discussions.rs
│   └── gitlab-cli/             # bin crate — thin clap adapter
│       └── src/
│           ├── main.rs
│           ├── cmd/            # one module per top-level command, mirrors resources
│           ├── output.rs       # stdout JSON/NDJSON emitters
│           ├── errout.rs       # stderr structured error emitter
│           └── raw.rs          # `api` escape hatch
├── tests/
│   ├── fixtures/               # sanitized 14.0.5 responses
│   ├── http_tests.rs           # wiremock-backed L2 tests
│   └── smoke.rs                # #[ignore]'d L3 tests against real instance
└── docs/
```

**Rationale**: keeping `gitlab-core` free of `clap`, stdout/stderr, and terminal concerns means we can later wrap it in an MCP server or embed it in another Rust program without refactoring.

**File size discipline**: no single source file should exceed ~300 lines; if a resource module approaches that, split by verb group (e.g., `merge_requests/read.rs` + `merge_requests/write.rs`).

### Dependency selection

| Crate | Purpose |
|---|---|
| `reqwest` (`rustls-tls`, `json`, `gzip`, `stream`) | HTTP client |
| `tokio` (`rt-multi-thread`, `macros`) | async runtime |
| `clap` (`derive`) | argument parsing |
| `serde` / `serde_json` | JSON (s)erialization |
| `thiserror` | error definitions |
| `tracing` + `tracing-subscriber` | structured stderr logging |
| `directories` | XDG/Windows config paths |
| `url` | URL building and percent-encoding |
| `governor` or hand-rolled token bucket | client-side RPS cap |
| `wiremock` *(dev-dep)* | mock HTTP server for L2 tests |
| `insta` *(dev-dep)* | snapshot tests for CLI output |

No use of `anyhow` in `gitlab-core` (library code — typed errors only). The `gitlab-cli` bin crate may use `anyhow` at the top level.

## 5. Command Tree

### Verb convention

Every resource supports the applicable subset of: `list`, `get`, `create`, `update`, `delete`. Resource-specific verbs are added only when GitLab has a dedicated endpoint (e.g., `mr merge`, `job retry`, `branch protect`).

### Top-level commands

```
gitlab project     list | get | create | update | delete | fork | archive | unarchive
gitlab group       list | get | create | update | delete | members | projects | subgroups
gitlab mr          list | get | create | update | close | reopen | merge | rebase |
                   approve | unapprove | diffs | commits | changes | pipelines
gitlab issue       list | get | create | update | close | reopen | link | unlink | move | stats
gitlab pipeline    list | get | create | retry | cancel | delete | variables
gitlab job         list | get | play | retry | cancel | erase | trace | artifacts
gitlab commit      list | get | create | diff | comments | statuses | cherry-pick | revert | refs
gitlab branch      list | get | create | delete | protect | unprotect
gitlab tag         list | get | create | delete | protect | unprotect
gitlab file        get | create | update | delete | blame | raw
gitlab repo        tree | archive | compare | contributors | merge-base
gitlab user        list | get | me | keys | emails
gitlab label       list | get | create | update | delete | subscribe | unsubscribe
gitlab note        list | get | create | update | delete
gitlab discussion  list | get | resolve | unresolve

# Utility
gitlab search      <scope> <query>
gitlab version
gitlab me
gitlab config      set-token | list | path

# Escape hatch
gitlab api <GET|POST|PUT|PATCH|DELETE> <path> [--data @file.json | '…'] [--query k=v]...
```

### Argument conventions

1. **Project identifier**: any command accepting a project accepts either a numeric ID (`123`) or a full path (`group/subgroup/project`). The CLI percent-encodes the path internally.
2. **Global flags**, available on every command:

   | Flag | Meaning |
   |---|---|
   | `--host <url>` | override host |
   | `--token <pat>` | override token |
   | `--output json\|ndjson` | default `json` |
   | `--limit N` | cap total records returned (list commands) |
   | `--no-paginate` | fetch only first page |
   | `--timeout <s>` | total request timeout |
   | `--retries N` | max retry count |
   | `--no-retry` | disable retries |
   | `--rps N` | client-side rate cap |
   | `--dry-run` | show the HTTP request but do not send |
   | `--yes` | skip confirmation prompt on writes |
   | `--verbose` | short for `GITLAB_VERBOSE=1` |

3. **Write safety**: `create/update/delete` and equivalents require `--yes` unless `GITLAB_ASSUME_YES=1` is set. `--dry-run` short-circuits before any network call and emits `{"dry_run": true, "method": …, "url": …, "body": …}` to stdout with exit code 10.

### Usage examples

```bash
# List every opened MR across a group, auto-paginated
gitlab mr list --group atoms/platform --state opened

# Comment on an MR (write → needs consent)
GITLAB_ASSUME_YES=1 gitlab note create \
  --project atoms/api --mr 123 --body "LGTM"

# Read a file at a specific ref
gitlab file get --project atoms/api --ref main --path src/foo.rs

# Escape hatch — pipeline schedules aren't in the 12 classes
gitlab api GET /projects/atoms%2Fapi/pipeline_schedules

# Stream 5 000 commits without loading them all in memory
gitlab commit list --project atoms/api --output ndjson --limit 5000 > commits.ndjson
```

## 6. Output Schema (stdout)

### Success shapes

- **list** commands: `stdout` = single JSON array with all pages concatenated. `--output ndjson` switches to one object per line; `--no-paginate` / `--limit N` cap the stream earlier.
- **get / create / update** commands: `stdout` = the JSON object GitLab returned, **unmodified**.
- **Action commands that return a body** (e.g., `mr merge`, `mr approve`, `job retry`, `pipeline retry`): `stdout` = the JSON object GitLab returned, unmodified. Same behavior as `get`.
- **Action commands returning 204 No Content** (e.g., `delete`, most `cancel`, `subscribe/unsubscribe`, `protect/unprotect`): `stdout` empty, exit 0. When `--output json` is explicitly passed, emit `{"ok": true, "status": 204}`.
- **Binary-returning commands** (`file raw`, `job artifacts`, `repo archive`, and `api` when the upstream `content-type` is not `application/json`): raw bytes are written verbatim to `stdout`. A stderr INFO log records the byte count and content-type so agents can confirm what they got.
- **`api` escape hatch**: parsed JSON passes through when `content-type: application/json`; any other content type falls into the binary path above.

### Field transparency

GitLab's raw field names are preserved. No renaming, no flattening, no `snake_case → camelCase`. README states clearly: "Output is the unmodified GitLab 14.0.5 REST response. Fields differ from 17.x."

### Pagination metadata

Never mixed into the stdout payload. When `--verbose` is on, page-fetch progress and final total are logged to stderr only.

## 7. Error Schema (stderr) & Exit Codes

### Error JSON (emitted to stderr on any non-success path)

```json
{
  "error": {
    "code": "not_found",
    "status": 404,
    "message": "404 Project Not Found",
    "request_id": "01KPD61BTWF4YKWEAXQ9RZ6FXR",
    "retryable": false,
    "details": {}
  }
}
```

- `code` is drawn from a fixed vocabulary:
  `invalid_args`, `unauthorized`, `forbidden`, `not_found`, `conflict`,
  `rate_limited`, `server_error`, `network`, `timeout`,
  `bad_request`, `unknown`.
  (Note: `--dry-run` is a success path, not an error, and exits 10 without emitting this struct.)
- `status` is the HTTP status, or `null` for purely-local errors (bad arguments, config unreadable, network hard-fail before response).
- `request_id` is copied from the `X-Request-Id` response header when present.
- `retryable` is the CLI's own judgement: `true` for 429 / 502 / 503 / 504 / connection resets.
- `details` carries GitLab's original body when the error body contained field-level validation data.

### Exit-code vocabulary

| Code | Meaning |
|---|---|
| 0 | Success (even on empty list) |
| 1 | Unknown error (catch-all) |
| 2 | Invalid arguments / clap parse failure |
| 3 | Authentication failure (401) |
| 4 | Authorization insufficient (403) |
| 5 | Resource not found (404) |
| 6 | Conflict / unprocessable (409, 422) |
| 7 | Rate limit exhausted after retries (429) |
| 8 | Server error after retries (5xx) |
| 9 | Network / timeout |
| 10 | `--dry-run` completed |

### Secrets masking

The `Display` / `Debug` impls for any struct that holds a PAT emit `glpat-****<last4>` only. A unit test asserts that the exact token string never appears in a rendered error or tracing span.

## 8. HTTP Layer & Reliability

### Client construction

- `reqwest::Client` built once per CLI invocation with:
  - `rustls-tls` (no system OpenSSL dependency → truly static binaries).
  - `connect_timeout(5s)`, `timeout(30s)` per request, `total_budget(120s)` across retries.
  - `gzip(true)`.
  - `User-Agent: gitlab-cli/<ver> (+rust; target=<triple>)`.
- `--tls-skip-verify` exists for self-signed dev instances; off by default; emits a stderr warning every time it's on.

### Authentication injection

`PRIVATE-TOKEN: <pat>` header on every request. Resolution order:
1. `--token <pat>`
2. `GITLAB_TOKEN` env var
3. `[host."<host>"] token = …` in TOML config
4. Fail with `code = unauthorized` if none found.

Config file must be `0600` on Unix; a relaxed mode emits a warning but does not abort.

### Retry policy

| Condition | Behavior |
|---|---|
| Connection refused / DNS / TCP reset / read timeout | 4 retries, backoff `0.5s → 1s → 2s → 4s` (with ±20 % jitter) |
| 5xx except 501 | Same as above |
| 429 | Honor `Retry-After` header verbatim; fallback to backoff series; 5 retries max |
| 4xx (except 429) | No retry — configuration/permission error |

Per-retry attempts are logged to stderr at `INFO`. `--no-retry` disables the policy entirely.

### Client-side throttle

Token-bucket cap, default `10 req/s` with burst `10`. `--rps N` overrides; `GITLAB_RPS` env var also honored. Purpose: when an agent fan-outs many `gitlab` invocations in parallel, keep the 14.0.5 Rails workers from tipping over.

### Pagination

- `core::page::PagedStream<T>` implements `Stream<Item = Result<T, GitlabError>>`.
- Per-page `per_page=100` is always requested.
- `Link` header parsed for `rel="next"`.
- CLI consumes the stream until `--limit N` satisfied, stream exhausted, or `--no-paginate` (single page).

### Observability

- `tracing_subscriber::fmt().with_writer(io::stderr)` — all logs on stderr.
- Default level `WARN`. `--verbose` or `GITLAB_VERBOSE=1` raises to `INFO`. `GITLAB_VERBOSE=debug` to `DEBUG`.
- Each CLI invocation assigns a local `invocation_id` (UUID v7). Every span and every error carries it, allowing agents dispatching N parallel calls to correlate.

### Concurrency model

The CLI itself issues requests serially. Async is internal plumbing only. No `--parallel` flag — agents should fan out via shell / their scheduler if they want parallelism, and let the client-side throttle keep total RPS safe.

## 9. Configuration & Distribution

### Config file

`~/.config/gitlab-cli/config.toml` (Linux/macOS) / `%APPDATA%\gitlab-cli\config.toml` (Windows):

```toml
default_host = "gitlab.deepwisdomai.com"

[host."gitlab.deepwisdomai.com"]
token = "glpat-…"
tls_skip_verify = false
rps = 10
default_project = "atoms/api"

[host."gitlab.com"]
token = "glpat-…"
```

Config-writing commands:
- `gitlab config set-token --host <h>` (prompts without echo, persists).
- `gitlab config list` (lists hosts, tokens masked).
- `gitlab config path` (prints the resolved path — useful in scripts).

Hand-editing TOML is supported and documented.

## 10. Testing Strategy

### Layer 1 — unit tests (alongside source)

- `parse_link_header`: valid `rel=next|prev|last`, malformed, missing, multi-relation.
- `auth::resolve`: full precedence matrix.
- `error::classify`: HTTP status × body shape → `(code, exit_code, retryable)`.
- `retry::backoff`: deterministic sequence under a seeded RNG.
- clap via `Command::try_parse_from`: missing required args, mutually-exclusive flags, env-var fallback.

### Layer 2 — HTTP behavior (wiremock-backed)

- One golden-path per resource: `list → get → create → update → delete`.
- Retry sequences: fixture `[429, 500, 200]` confirms final success + expected request count + backoff timing.
- Pagination: fixture of 3 pages with `Link` headers, assert `PagedStream` produces the correct total.
- Error mapping: fixtures for 401, 403, 404, 409, 422, 429 (with and without `Retry-After`), 500, 502, 503, 504.
- Secrets masking: assert `glpat-*` literal appears in no log, error, span, or snapshot.
- `api` escape hatch: GET/POST/PUT/PATCH/DELETE round-trip with headers and body preserved.

### Layer 3 — smoke tests against real 14.0.5-ee (opt-in)

- `tests/smoke.rs`, `#[ignore]` by default.
- Enabled when `GITLAB_TEST_HOST`, `GITLAB_TEST_TOKEN`, `GITLAB_TEST_PROJECT` are all set.
- **Read-only** only: `version`, `me`, `project get`, `mr list --limit 5`, `file get`.
- No write operations — we do not want CI to mutate a real instance.

### Fixtures

- `tests/fixtures/*.json` — responses captured from a real 14.0.5 instance, with tokens, emails, and internal usernames sanitized.
- A fixture-regeneration script (`scripts/record-fixtures.sh`) is documented but not run in CI.

### CI matrix (GitHub Actions)

| Step | Command |
|---|---|
| Format | `cargo fmt --check` |
| Lint | `cargo clippy --all-targets -- -D warnings` |
| Test | `cargo test --all` (L1+L2) |
| Supply chain | `cargo deny check` |
| Release build | `cargo build --release --target <target>` per target |

### Coverage

Measured with `cargo-llvm-cov`, goal ≥ 80 % for `gitlab-core`, ≥ 70 % for `gitlab-cli`. No hard CI gate — correctness trumps percentage.

### TDD discipline

Implementation work follows `superpowers:test-driven-development`:
1. Write one failing wiremock test.
2. Minimal code to pass.
3. Refactor.
4. Repeat per resource module.

## 11. Risks & Open Questions

- **GitLab 14.0.5 field drift**: some fields documented in the tag's `doc/api` may have been added by patches we haven't read. Fixture-based tests against a real instance are the primary defense.
- **EE-only endpoints**: `merge_request_approvals` and epics require EE. The instance is EE but the license status may change; CLI returns a clean `code = forbidden` with the original GitLab message intact if a license lapse occurs.
- **Rate-limit reality**: 14.0.5 defaults to 600 req/min authenticated, but operators often tighten this. The 10 rps client cap is deliberately under that; we can raise it via `--rps`.
- **Security posture of the instance**: 14.0.5 is EOL with multiple CVEs (CVE-2022-2185 etc.). Out of scope for this CLI, but noted for the user to drive an upgrade conversation separately.

## 12. Out of Scope (explicitly)

- Multi-version compatibility layer.
- GraphQL client.
- OAuth / CI_JOB_TOKEN / Impersonation token authentication.
- Interactive TUI, shell completions, colorized prose output.
- Admin endpoints (`/admin/*`), audit events, Geo nodes, system hooks.
- Epic / iteration / vulnerability / merge-train commands (deferred; reachable via `api` escape hatch).

## 13. Acceptance Criteria (for MVP)

The MVP is "done" when every item below is true:

1. All 15 top-level commands + `api` escape hatch parse correctly under `cargo test` and `--help` output is consistent.
2. Against a locally-mocked GitLab (wiremock), each of the 12 resource families passes its golden-path `list → get → create → update → delete`.
3. A single static binary builds for macOS arm64 + Linux musl x86_64 and runs a smoke test (`version`, `me`, `project get`) against the real `gitlab.deepwisdomai.com` instance using a developer PAT.
4. Running the same smoke with a deliberately broken PAT exits with code 3 and emits the structured error JSON on stderr.
5. `mr list --group <g>` with 250+ results returns all rows (verified count) in a single JSON array without manual paging.
6. `--dry-run` on every `create/update/delete` does not emit an HTTP request (verified by wiremock assertion) and exits 10.
7. README documents: command list, env vars, config TOML, exit codes, "targets GitLab 14.0.5-ee, no version negotiation".

## Appendix A — Command-to-API mapping (abbreviated)

Full mapping will live in source as doc-comments on each resource module. The sample below shows the shape:

```
mr list    → GET /projects/:id/merge_requests          (pagination: offset)
mr get     → GET /projects/:id/merge_requests/:iid
mr create  → POST /projects/:id/merge_requests
mr update  → PUT /projects/:id/merge_requests/:iid
mr merge   → PUT /projects/:id/merge_requests/:iid/merge
mr approve → POST /projects/:id/merge_requests/:iid/approve   (EE)
mr diffs   → GET /projects/:id/merge_requests/:iid/diffs
…
```
